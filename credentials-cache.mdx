---
title: "Reduce UAC popups with gsudo credentials cache"
description: "Learn how the gsudo credentials cache works, when to use each cache mode, and how to balance convenience against security trade-offs."
---

The credentials cache lets you approve elevation once with a UAC popup and then run multiple elevated commands without seeing another popup — until the cache expires or you end the session. It is opt-in because it carries a security trade-off: if a malicious process is already running on your machine, it could use [DLL injection](https://en.wikipedia.org/wiki/DLL_injection#Approaches_on_Microsoft_Windows) to force the open cache session to elevate silently without your knowledge.

## How it works

When you start a cache session, gsudo launches a background elevated process that acts as a broker. Subsequent `gsudo` invocations from the same parent process communicate with that broker instead of triggering a new UAC popup. The session closes automatically after it has been idle for 5 minutes (configurable via `CacheDuration`), or when the parent process exits.

<Warning>
  The credentials cache is safe only when you are confident there are no malicious processes running on your system. An attacker who can inject code into your shell process could silently trigger elevations through an open cache session. Understand this trade-off before enabling cache mode.
</Warning>

## Cache modes

Set the cache mode once and it persists across sessions:

```powershell
gsudo config CacheMode {Auto|Explicit|Disabled}
```

<AccordionGroup>
  <Accordion title="Explicit (default)" defaultOpen={true}>
    Every elevation shows a UAC popup unless you first start a cache session manually with `gsudo cache on`. This gives you full control over when silent elevation is possible.

    ```powershell
    # Explicit is the default — this line is optional:
    gsudo config CacheMode Explicit

    gsudo command1          # UAC popup #1
    gsudo command2          # UAC popup #2
    gsudo cache on          # UAC popup #3 — cache session starts
    gsudo command3          # no popup
    gsudo command4          # no popup
    gsudo cache off         # ends the cache session
    gsudo command5          # UAC popup again
    ```

    Use `gsudo cache on` when you know you are about to run several elevated commands in a row, then `gsudo cache off` (or `gsudo -k`) when you are done.
  </Accordion>

  <Accordion title="Auto">
    The first elevation in a session triggers a UAC popup and automatically starts a cache session. Subsequent elevations are silent until the cache expires or you end it — this is the closest behavior to Unix sudo.

    ```powershell
    # Run once to enable — this setting persists:
    gsudo config CacheMode Auto

    gsudo command1          # UAC popup — cache session starts automatically
    gsudo command2          # no popup
    gsudo command3          # no popup
    gsudo cache off         # ends the cache session; next elevation will prompt again
    ```

    <Tip>
      Auto mode is the most convenient option for day-to-day use if you understand and accept the security trade-off.
    </Tip>
  </Accordion>

  <Accordion title="Disabled">
    Every elevation shows a UAC popup with no exceptions. Attempts to start a cache session with `gsudo cache on` return an error. Use this mode in high-security environments where you want to ensure every privilege escalation is explicitly confirmed.

    ```powershell
    gsudo config CacheMode Disabled

    gsudo command1          # UAC popup — no cache possible
    gsudo cache on          # error: cache is disabled
    ```
  </Accordion>
</AccordionGroup>

## Starting and stopping a cache session

```powershell
gsudo cache on  [-p {pid}] [-d {duration}]   # start a cache session
gsudo cache off [-p {pid}]                   # stop a cache session
gsudo -k                                     # stop ALL cache sessions immediately
```

### Options

<AccordionGroup>
  <Accordion title="-p | --pid {pid}">
    Specifies which process is allowed to use the cache session. The default is the calling process's PID, which means only the shell that ran `gsudo cache on` can elevate silently. Pass `0` to allow any process from the current user to use the cache.

    ```powershell
    gsudo cache on -p 0        # allow any process from this user
    gsudo cache on -p 1234     # allow only process with PID 1234
    ```
  </Accordion>

  <Accordion title="-d | --duration {hh:mm:ss}">
    Sets how long the cache session can stay idle before it closes automatically. Overrides the `CacheDuration` setting for this session only. Use `-1` to keep the session open indefinitely (until you close it with `gsudo cache off` or `gsudo -k`, or until you log off).

    ```powershell
    gsudo cache on -d 00:30:00   # 30-minute idle timeout
    gsudo cache on -d -1         # no auto-expiry for this session
    ```
  </Accordion>

  <Accordion title="-s | --sid {sid}">
    Specifies a Windows Security Identifier (SID) to restrict cache access to a specific user by SID rather than by PID. This is an advanced option used in scripted environments where the calling PID is not reliable.

    ```powershell
    gsudo cache on --sid S-1-5-21-...
    ```
  </Accordion>
</AccordionGroup>

## Configuring the default cache duration

The `CacheDuration` setting controls how long a cache session stays open when idle. The default is 5 minutes (`00:05:00`). This applies to all new sessions unless you override it with `-d` at the time you run `gsudo cache on`.

```powershell
gsudo config CacheDuration 00:15:00   # change default to 15 minutes
gsudo config CacheDuration Infinite   # sessions never expire automatically
gsudo config CacheDuration --reset    # restore the 5-minute default
```

## Ending all cache sessions

Use `gsudo -k` to immediately terminate every active cache session for your user. Run this before stepping away from your computer or handing it to someone else.

```powershell
gsudo -k
```

## Security considerations

<CardGroup cols={2}>
  <Card title="When the cache is safe" icon="shield-check">
    The cache is safe when you are the only user of your machine and you are confident no malicious software is running. The cache session automatically limits itself to the process (or PID) that started it.
  </Card>
  <Card title="When to avoid the cache" icon="shield-exclamation">
    Avoid leaving an open cache session when you share your machine, when you are on an untrusted network, or when running software from untrusted sources. Use `gsudo -k` before walking away.
  </Card>
</CardGroup>

A malicious process with access to your shell (for example through DLL injection into `cmd.exe` or `powershell.exe`) could silently instruct an open gsudo cache session to elevate arbitrary commands. This is the same fundamental risk present in Unix sudo — it is a deliberate trade-off between convenience and security. gsudo makes the cache opt-in (defaulting to `Explicit` mode) so you consciously choose to accept this trade-off.
